# 9. Usage Patterns and Examples

This chapter turns the protocol into application shapes.

## Pattern 1: Simple Dataset Fetch

Use when:

- dataset is stable,
- a `PATH` descriptor makes sense,
- server returns one or more endpoints.

Flow:

```text
descriptor(PATH) -> GetFlightInfo -> endpoint[] -> DoGet(ticket) -> RecordBatch stream
```

Rust sketch:

```rust
use arrow_flight::{FlightClient, FlightDescriptor};
use futures::TryStreamExt;
use tonic::transport::Channel;

async fn fetch_table() -> arrow_flight::error::Result<()> {
    let channel = Channel::from_static("http://127.0.0.1:50051").connect().await.unwrap();
    let mut client = FlightClient::new(channel);

    let info = client
        .get_flight_info(FlightDescriptor::new_path(vec!["db".into(), "table".into()]))
        .await?;

    let ticket = info.endpoint[0].ticket.clone().expect("endpoint ticket");
    let batches = client.do_get(ticket).await?.try_collect::<Vec<_>>().await?;
    assert!(!batches.is_empty());
    Ok(())
}
```

Why this pattern works well:

- clean separation between discovery and fetch,
- schema available before transfer,
- easy to scale later into multiple endpoints.

## Pattern 2: Query Planning Then Fetch

Use when:

- request is dynamic,
- the dataset is generated by a command or query.

Here `CMD` descriptors are the right abstraction. The server may parse SQL, build a plan, or allocate a result handle before returning endpoints.

Why not go directly to `DoGet`:

- query planning may need time,
- fetch tickets may be partition-specific,
- clients may want schema and totals first.

## Pattern 3: Long-Running Query with Polling

Use when:

- planning or execution is slow,
- partial endpoints may appear over time.

Flow:

```text
poll_descriptor0
  -> PollInfo(partial info, next descriptor)
  -> fetch currently available endpoints
  -> poll again with next descriptor
```

This allows overlap between:

- server-side work,
- client-side data consumption.

That is a big throughput/latency win in distributed systems.

## Pattern 4: Batch Upload with `DoPut`

Use when:

- client needs to stream Arrow batches into the server,
- server may return acknowledgements or ingest results.

Rust sketch:

```rust
use arrow_array::{RecordBatch, UInt64Array};
use arrow_flight::encode::FlightDataEncoderBuilder;
use arrow_flight::FlightClient;
use futures::stream;
use std::sync::Arc;

async fn upload(mut client: FlightClient, batch: RecordBatch) -> arrow_flight::error::Result<()> {
    let outbound = FlightDataEncoderBuilder::new()
        .build(stream::iter(vec![Ok(batch)]));

    let _put_results = client.do_put(outbound).await?;
    Ok(())
}
```

Behind the scenes this yields:

- schema message,
- optional dictionary messages,
- batch messages,
- server `PutResult` stream.

## Pattern 5: Full-Duplex Exchange

Use when:

- request and response streams influence each other during the session,
- you need a conversational dataflow.

Examples:

- interactive execution engines,
- pushdown filters with streamed responses,
- custom compute sessions.

Use `DoExchange`, not a fake `DoPut`+`DoGet` pair, when duplex semantics are real.

## Pattern 6: Metadata and Administration via Actions

Use `DoAction` and `ListActions` for operations that are neither dataset discovery nor bulk data transfer:

- cancel a query,
- create prepared statements,
- lifecycle or maintenance commands.

Benefits:

- transport stays structured,
- clients can introspect supported actions,
- management semantics are not overloaded into fake data fetches.

## Pattern 7: Flight SQL

Use when:

- your application semantics are relational,
- you want standardized SQL metadata, prepared statements, and transaction operations,
- you still want Arrow-native streaming transport.

Study `arrow-flight/examples/flight_sql_server.rs` as the practical example.

## Pattern 8: Simple In-Memory Server

For tests, local demos, or lightweight services:

- implement `FlightService`,
- use `batches_to_flight_data` from `utils.rs`,
- return boxed streams of `FlightData`.

This is enough for a correct minimal server, but not usually enough for a production one.

## Choosing the Right RPC

```text
Need                                      Use
---------------------------------------   -----------------
List available datasets                   ListFlights
Resolve one dataset/query                 GetFlightInfo
Poll long-running query                   PollFlightInfo
Fetch data                                DoGet
Upload data                               DoPut
Bidirectional session                     DoExchange
Run control/management operation          DoAction
Discover supported management ops         ListActions
Learn only the schema                     GetSchema
Authenticate / establish session          Handshake
```

## Source Markers

- `arrow-flight/src/client.rs`: base client examples for `do_get`, `do_put`, `do_exchange`, `do_action`
- `arrow-flight/examples/server.rs`: base server skeleton
- `arrow-flight/examples/flight_sql_server.rs`: richer server pattern
- `arrow-flight/src/utils.rs`: small-server helpers
- `arrow-flight/tests/client.rs`: usage-level behavior checks

## Practical Rule

When designing a Flight application, first choose the correct conversation shape. Only then choose the specific Arrow payloads. Most bad Flight designs start by shoving everything into `DoGet` or `DoAction` without asking what interaction pattern they actually need.
