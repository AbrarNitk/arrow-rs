# 2. Message Model and FlatBuffer Metadata

The most important IPC concept is that a message is split into:

- a flatbuffer metadata message,
- and a body of raw Arrow buffers.

If you keep that separation sharp, the rest of the format becomes much easier to reason about.

## The Generic `Message`

The generated types in `arrow-ipc/src/gen/Message.rs` are re-exported by `arrow-ipc/src/lib.rs`, and the writer builds them in `arrow-ipc/src/writer.rs`.

Conceptually, a top-level IPC message contains:

```text
Message
├── version
├── header_type
├── header
└── bodyLength
```

The `header_type` decides how to interpret `header`.

In ordinary Arrow IPC data exchange, the interesting message kinds are:

- `Schema`
- `RecordBatch`
- `DictionaryBatch`

## Why Schema, Batch, and Dictionary Are Separate Messages

This is not accidental.

They are separate because they answer different state questions:

- `Schema`: what is the shape of subsequent data?
- `DictionaryBatch`: what dictionary values are currently bound to a dictionary id?
- `RecordBatch`: what are the actual row-grouped values for this segment of data?

Collapsing these into one structure would make reuse and state updates much harder.

## `Schema` Metadata

Schema metadata is generated by `IpcSchemaEncoder` in `arrow-ipc/src/convert.rs`.

This encoder is responsible for mapping Arrow `Schema` and `Field` values into the flatbuffer schema model.

Important details:

- fields become nested flatbuffer `Field` tables,
- custom metadata is serialized explicitly,
- dictionary-encoded fields require dictionary ids,
- the schema includes endianness.

### Why Schema Is Flatbuffer Metadata Instead of Body Data

Because schema is small, structured, and frequently read before any batch body is touched.

It is exactly the kind of data FlatBuffers is good at:

- tree-shaped,
- richly typed,
- not performance-critical in bulk byte count.

## `RecordBatch` Metadata

Record batch metadata is built in `record_batch_to_bytes` in `arrow-ipc/src/writer.rs`.

The key pieces are:

- `length`: logical row count
- `nodes`: `FieldNode`s for the array tree
- `buffers`: ordered buffer descriptors
- optional body compression info
- optional `variadicBufferCounts`

Diagram:

```text
RecordBatch metadata
├── length
├── nodes[]
│   each node = { length, null_count }
├── buffers[]
│   each buffer = { offset, length }
├── compression?
└── variadicBufferCounts?
```

### Why `FieldNode`s Exist

The body buffers alone do not tell you array tree structure.

For nested data, the reader needs to know:

- where each logical field/child begins in the traversal,
- how many logical elements that node represents,
- how many nulls it claims to contain.

`FieldNode`s provide the logical shape that pairs with the physical buffers.

### Why `buffers[]` Exist

The body is just a concatenated byte region. The reader needs an index that says:

- which slices correspond to which logical buffers,
- in what order they appear,
- how long each is.

That is the role of `buffers[]`.

## `DictionaryBatch` Metadata

Dictionary messages are built by `dictionary_batch_to_bytes` in `arrow-ipc/src/writer.rs`.

They carry:

- dictionary id,
- embedded record-batch-style data for the dictionary values,
- an `isDelta` flag.

This is a strong design because dictionary values themselves are Arrow arrays. Reusing record-batch-like metadata avoids inventing a new payload description just for dictionaries.

## `bodyLength`

`bodyLength` exists on the outer `Message` because the metadata and body are separate regions.

Why this matters:

- the reader can locate the exact body region for this message,
- stream readers can skip efficiently,
- file readers can interpret blocks correctly,
- padding/alignment can be preserved without ambiguity.

## The Metadata/Body Contract

Think of one IPC message as:

```text
framing prefix
  -> tells you how long the flatbuffer metadata is

flatbuffer Message
  -> tells you what kind of message this is
  -> tells you how large the body is
  -> tells you how to interpret the body buffers

body bytes
  -> actual Arrow buffers
```

That is the entire contract in one diagram.

## Schema Conversion Round-Trip

`arrow-ipc/src/convert.rs` provides both directions:

- `IpcSchemaEncoder`
- `fb_to_schema`

This is important for more than convenience. It keeps the mapping logic centralized so:

- writers and readers use the same type translation rules,
- flatbuffer details stay localized,
- schema evolution bugs are less likely to fork between encode and decode paths.

## Source Markers

- `arrow-ipc/src/convert.rs`: `IpcSchemaEncoder`, `fb_to_schema`, metadata conversion
- `arrow-ipc/src/writer.rs`: `schema_to_bytes_with_dictionary_tracker`, `record_batch_to_bytes`, `dictionary_batch_to_bytes`
- `arrow-ipc/src/lib.rs`: generated schema/message re-exports

## Implementation Lesson

If you were rebuilding IPC, you would want to preserve this exact responsibility split:

- schema conversion code,
- message metadata builders,
- body buffer writers,
- framing code.

Merging them all into one monolithic serializer would make the invariants much harder to see and maintain.
